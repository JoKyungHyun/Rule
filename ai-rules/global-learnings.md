# 공통 개발 경험 (Global Learnings)

## API 설계

- API 응답은 항상 `{ data, error }` 구조를 사용한다
- HTTP 상태 코드를 명확하게 사용한다
- 사용자용 에러 메시지와 내부 에러 로그는 구분한다

## 프론트엔드

- 재사용 가능한 컴포넌트를 우선한다
- 모바일 우선 설계를 적용한다
- 불필요한 전역 상태 사용을 지양한다

## 백엔드

- 모든 입력값은 검증한다
- `async/await` 패턴을 일관되게 사용한다
- 비즈니스 로직과 컨트롤러를 분리한다

## 데이터베이스

- ORM 사용 시 N+1 쿼리를 주의한다
- 인덱스가 필요한 컬럼은 명시적으로 설정한다
- 마이그레이션 파일은 되돌릴 수 있도록 down 스크립트도 작성한다
- 트랜잭션이 필요한 작업은 반드시 트랜잭션으로 묶는다

## 성능

- API 응답 시간은 300ms 이하를 목표로 한다
- 불필요한 API 호출을 줄인다
- 필요 시 캐싱을 적용한다

## 코드 품질

- 함수는 단일 책임 원칙을 따른다
- 변수명은 의미 있게 작성한다
- 중복 코드를 제거한다 (DRY 원칙)

## 환경 설정

- 환경별 설정은 `.env` 파일로 분리한다 (`.env.local`, `.env.production` 등)
- `.env` 파일은 절대 버전 관리에 포함하지 않는다
- 환경 변수 목록은 `.env.example`로 문서화한다

## 개발 원칙

- 기존 프로젝트 패턴을 우선 따른다
- 새로운 라이브러리는 필요할 때만 도입한다
- 라이브러리 도입 시 번들 크기와 유지보수 상태를 확인한다
